# ERC-4337 Research Plan

## Purpose

Plan research into interactions among ERC-4337, MIP-4, and EIP-7702 delegated EOAs without building the experiment yet.

## Current Verified Facts

- A protocol-created EIP-7702 delegated EOA in a real Monad Testnet authorization-list transaction produced `dippedIntoReserve() == true` while its balance moved from 19 MON to 9 MON.
- The successful transaction was sponsor-submitted.
- Threshold-boundary and sender-versus-sponsor behavior remain unverified.
- No ERC-4337-style execution path has been tested in this repository.

## System Map

| Component | Research role |
| --- | --- |
| `UserOperation` | Encodes the account action that may change the delegated EOA balance |
| `EntryPoint` | Executes and sequences operations in a bundle |
| Bundler | Submits the bundle transaction |
| Paymaster | May sponsor gas and alter balance-flow assumptions |
| Delegated EOA | Candidate account whose reserve state is observed |
| Reserve-balance violation | State transition under investigation |
| `dippedIntoReserve()` | Observation mechanism to sample at defined checkpoints |

Core question:

```text
Can an ERC-4337-style operation involving a delegated EOA cause dippedIntoReserve() == true,
and can that violation be detected or isolated inside a bundle?
```

## Open Questions

- Which address is classified as the transaction sender, sponsor, caller, and affected account at each layer?
- Can one operation's reserve state be observed by another operation in the same bundle?
- Does a paymaster-funded flow change the relevant balance transition?
- Where should observations occur to distinguish operation-local and transaction-wide state?

## Next Steps

- Complete threshold-boundary and sender/sponsor experiments first.
- Define a minimal EntryPoint-style call graph and observation checkpoints on paper.
- Specify control operations, expected logs, balance accounting, and failure isolation.
- Keep implementation blocked until the prerequisite evidence is available.
